# Implementation Plan: Auto-Refresh Site Data

**Branch**: `015-auto-refresh` | **Date**: 2026-03-05 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/015-auto-refresh/spec.md`

## Summary

Add a background polling module to the site that checks for data and code changes every 30 seconds. When data files change, the current view re-renders in place without a page reload. When front-end code changes (detected via a generated version file), a full page reload occurs. The polling is invisible to users -- no blinks, flashes, or layout shifts.

**Technical approach**: Cache-busting fetch + JSON string comparison for data; a build-generated `site-version.txt` for code change detection. The existing render/bind component pattern supports re-invocation without modification.

## Technical Context

**Language/Version**: Vanilla JavaScript ES modules (frontend); TypeScript 5.x on Node.js 20+ (build scripts, E2E tests)
**Primary Dependencies**: None (pure DOM, no framework) -- no new dependencies added
**Storage**: JSON data files (`data/current.json`, `data/history.json`, `data/feature-flags.json`) -- read-only from frontend, no changes to data format
**Testing**: Playwright (E2E), Jest (unit) -- existing setup
**Target Platform**: Modern browsers (Chrome, Firefox, Safari, Edge)
**Project Type**: Static web application (GitHub Pages)
**Performance Goals**: Refresh check completes in <1s; re-render imperceptible to user
**Constraints**: No server-side capabilities (static hosting); polling interval 30s; no visible UI artifacts during checks
**Scale/Scope**: Single-page dashboard with 5 routes; 4 data files; ~1,400 lines of frontend JS
